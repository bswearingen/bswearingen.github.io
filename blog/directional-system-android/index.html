May 2, 2017
3 minutes read

I haven’t tested this very thoroughly, so this post is going to mostly stick to generalities.

My goal for the “direction” part of the backpack direction system was to do as little work as possible. The interesting parts of this project, for me, were in conveying those directions to a user. The actual challenge of getting directions from point A to point B isn’t one that I have much interest in solving. With that in mind, I thought that the best solution was to piggy-back on another service that already finds and provides turn-by-turn directions. It seemed that the easiest way to do this would be to read the notifications that Google Maps uses for turn-by-turn navigation.

The first potential solution that I came across was this tutorial: Building Accessibility Services. It provides a detailed explanation of what an Android accessibility service is:

An accessibility service is an application that provides user interface enhancements to assist users with disabilities, or who may temporarily be unable to fully interact with a device. For example, users who are driving, taking care of a young child or attending a very loud party might need additional or alternative interface feedback.

This seemed to perfectly describe what was planned for the direction system - providing haptic feedback in a situation where I’m unable to receive visual or audio. But reading some of the other documentation on accessibility services in Android, specifically the AccessibilityService reference documentation, describe them as something that “should only be used to assist users with disabilities in using Android devices and apps.” I chose to ignore this and stick with the first definition.

After deciding to use an accessibility service to handle the piggy-backing, the next decision was whether to use Google Maps as originally planned. This ended up posing a bit of a problem. The notifications that Google Maps provides don’t seem to provide an easily discoverable way to parse whether they’re indicating a left turn or a right turn. The notifications use images to indicate left and right turns, and there doesn’t seem to be a way to easily differentiate them from their IDs (if you have a method of doing that, please let me know).

Luckily another turn-by-turn direction application, OsmAnd, provides notifications that are easily to parse.

With the easy to find direction and distance to the turn in the new notifications, creating an accessibility service that parsed OsmAnd notifications as they appeared and then sent a message to the backpack over Bluetooth was reasonably simple.

The complete project is here, but it’s the complete Android Studio project, so it’s a bit of a mess. At some point I’ll look into the best practices for storing Android Studio projects on version control, but for now it’s there in all its mess.
